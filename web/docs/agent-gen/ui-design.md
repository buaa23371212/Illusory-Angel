# 网页设计文档

## 目标平铺设计推荐

### 设计原则

默认采用平铺设计是因为目标作为项目的基本组成单元，平铺展示能够保持信息的扁平化呈现，用户可以一目了然地看到所有目标，减少认知负担。

### 布局方式

采用卡片式布局，每个目标占据独立卡片。卡片展示目标的核心信息，包括标题、描述、状态、进度等关键数据。卡片之间保持适当间距，保证视觉清晰度。

### 响应式适配

采用流式网格布局，根据屏幕宽度自动调整列数。大屏幕上显示多列，小屏幕自动变为单列布局，保证在各种设备上都有良好的可读性。

### 排序与筛选

平铺设计保留基础排序功能，支持按创建时间、优先级、完成状态等维度排序。同时提供基础筛选能力，用户可以快速过滤出特定状态的目标。排序和筛选保持在页面顶部，不占用过多垂直空间。

### 交互设计

支持点击卡片进入详情视图，详情视图展示更完整的信息和操作入口。卡片本身支持常用快捷操作，比如标记完成、快速编辑等，提升操作效率。

### 视觉层次

通过卡片阴影、圆角、内边距建立清晰的视觉层次。不同状态的目标使用不同的视觉标识，完成的目标使用弱化处理，进行中的目标突出显示，帮助用户快速识别。

## 插件化UI扩展设计

### 设计理念

核心系统默认使用平铺展示，当安装了支持前置目标组织的插件时，插件可以提供自定义的UI展示方式。这种设计保持了核心的简洁性，同时允许插件根据自身需求优化展示效果。

### 插件UI注册机制

插件需要在初始化时向核心系统注册自己的UI组件。注册信息包含插件标识、支持的展示类型、组件入口等元数据。核心系统维护一个插件UI注册表，根据当前配置决定使用哪种展示方式。

### 数据接口约定

插件UI组件遵循统一的数据接口规范，从核心系统获取目标数据。核心系统负责数据获取和持久化，插件只负责展示逻辑和交互逻辑。这种分工保证了数据一致性，插件无需关心数据存储细节。

### 运行时切换机制

页面加载时，核心系统检查当前项目是否启用了相关插件。如果插件存在且启用，使用插件提供的UI组件渲染目标列表；如果没有对应插件，回退到默认的平铺展示。切换对用户透明，无需手动干预。

### 通信协议

核心系统与插件UI组件通过预定义的事件接口进行通信。当用户在插件UI中操作目标时，事件通过核心系统统一处理，保证数据更新逻辑的一致性。插件只需要触发事件，不需要关心后续的数据处理流程。

### 样式隔离

插件UI组件需要保持样式隔离，避免影响核心系统的其他部分。插件使用独立的样式命名空间，防止样式冲突。核心容器为插件UI提供独立的渲染上下文，保证样式不会相互污染。

### 可扩展性

设计预留多种展示类型的扩展可能，不限于树形结构。未来不同插件可以实现不同的组织方式，比如看板、时间线、思维导图等。核心系统不限制插件的展示形式，只提供统一的接入点。

### 用户体验

切换到插件UI后，保留核心功能的入口位置不变，用户无需重新适应界面布局。导航、搜索、筛选等基础功能保持在原有位置，只改变目标列表的展示方式。

### 降级方案

如果插件被禁用或卸载，系统自动无缝切换回默认平铺展示，用户数据不会丢失。所有目标数据仍然完整保留，只是展示方式回归默认，保证系统的鲁棒性。

### 预先预留扩展点

在核心代码开发阶段，就需要为插件自定义UI预留接入位置。不能等到开发完插件后才修改核心代码，那样会增加修改成本和引入bug的风险。

### 可能的扩展点

系统需要预先预留多个UI扩展点，覆盖不同层级的定制需求。

目标列表展示区域是最主要的扩展点，允许插件完全替换目标的组织和展示方式。

目标卡片内容区域允许插件在默认卡片中插入额外的UI元素，展示插件特定信息，不改变整体布局。

项目详情页侧边栏允许插件添加自定义面板，展示插件相关的统计信息或操作入口。

导航栏允许插件注册自定义菜单项，跳转到插件提供的页面。

项目操作菜单允许插件添加自定义操作选项，执行插件特定的功能。

目标右键菜单允许插件添加上下文相关的操作项。

表单字段区域允许插件在创建或编辑目标时插入自定义表单项，收集插件需要的数据。

详情页底部允许插件添加自定义内容区域，展示扩展信息。

页面头部操作区允许插件添加自定义操作按钮，快速访问插件功能。

### 组件化接入设计

目标展示区域设计为一个容器组件，这个容器组件负责判断应该渲染哪种UI。容器内部根据插件注册表的信息，动态选择渲染默认平铺组件还是插件注册的自定义组件。这种设计从一开始就把扩展点建立好，后续插件只需要注册即可使用，不需要修改核心代码。

### 依赖注入方式

核心代码不直接导入插件组件，而是通过依赖注入的方式从插件注册表获取。这样核心代码和插件代码没有编译时依赖，插件可以独立开发和部署，不会影响核心构建流程。

### 接口先行开发

在开发核心功能时，就定义好插件UI需要实现的接口契约。包括props结构、事件回调格式、上下文数据结构等。所有组件无论是默认还是自定义，都遵循同一个接口。这样后续插件开发只需要实现接口即可，不需要调整核心代码。

### 默认实现优先

核心系统总是提供默认实现，保证在没有插件时系统功能完整可用。插件只是替换展示部分，核心的数据处理逻辑仍然由核心系统负责。这种分工使得即使没有插件，核心体验依然完整，插件只提供优化体验。

### 对开发成本的影响

如果开发时就预留了扩展点，后续添加插件自定义UI非常简单，只需要插件注册组件，核心代码不需要改动，修改成本很低。如果没有预先留扩展点，等到开发插件时再重构核心代码，不仅改动范围大，还可能破坏现有功能，测试成本也会大幅增加。因此预先预留扩展点是更低成本的做法。
