# 游戏养成计划器 - 开发指南

## 项目概述

游戏养成计划器是一个帮助用户规划和管理游戏养成目标的工具。用户可以创建项目，在每个项目中定义多个养成目标，并跟踪这些目标的进度。

## 核心数据结构设计

### 整体架构

项目采用简洁层级数据结构：
- **项目（Project）** - 最高层级组织单元，包含多个目标，也可以有约束（全局约束）
- **目标（Goal）** - 具体的养成目标，隶属于某个项目，可以嵌套依赖
- **约束（Constraint）** - 附加在项目或目标上的条件限制

### 项目（Project）

#### 核心属性

- **ID**：唯一标识符，用于索引和关联
- **名称**：项目名称，用户自定义
- **描述**：项目简介说明
- **创建时间**：记录项目创建时间
- **更新时间**：记录最后修改时间
- **状态**：活跃/已完成/已存档
- **排序权重**：支持用户自定义排序

#### 关系

一个项目可以包含 **零个或多个** 目标。  
一个项目可以有 **零个或多个** 约束（全局约束）。

**设计思路**：项目级别约束用于定义全局资源掉落限制，例如"每日宝石掉落120个"，这样所有需要该资源的目标都能共享这个配置。

### 目标（Goal）

#### 核心属性

- **ID**：唯一标识符
- **所属项目ID**：关联到父项目
- **名称**：目标名称
- **描述**：目标详细说明
- **类型**：养成类型（装备制作/材料收集/等级提升/角色培养/宠物培养/成就解锁/资源获取等）
- **所需数量**：目标总需求量（如收集300个碎片）
- **当前拥有数量**：用户已完成/已拥有数量
- **优先级**：极高/高/中/低/极低，支持多级优先级
- **状态**：未开始/进行中/已完成/已暂停
- **创建时间**：目标创建时间
- **更新时间**：最后修改时间
- **预计完成时间**：预估完成时间（可自动计算）
- **实际完成时间**：实际完成时间戳
- **标签**：自定义标签数组，用于分类和筛选
- **备注**：额外说明信息

#### 关系

- 一个目标必须 **属于且仅属于** 一个项目
- 一个目标可以依赖 **零个或多个** 其他目标（作为前置材料）
- 一个目标可以有 **零个或多个** 约束

**设计思路**：材料需求本身也是一个独立目标。例如，"收集300个氩级武器碎片"本身就是一个目标，"制作氩级武器"依赖于这个目标。

### 约束（Constraint）

约束是附加在目标上的条件限制。

#### 核心属性

- **ID**：唯一标识符
- **所属目标ID**：关联到目标
- **类型**：约束类型
  - 依赖约束：需要依赖的目标已完成（或达到一定数量）
  - 属性约束：依赖目标需要满足特定属性条件（如战斗力 > X）
  - 比较约束：A目标的某属性 > B目标的某属性
  - 数量约束：多个依赖目标都需要满足
- **参数**：约束参数（JSON格式存储，根据类型不同而不同）
- **描述**：约束说明

#### 常见约束类型示例

**依赖约束**：制作氩级武器需要收集300碎片和300万能石 → 添加两个依赖约束，分别指向"收集氩级武器碎片"和"收集万能石"两个目标。

**属性约束**：需要素材满足满战 → 添加属性约束，要求目标属性战斗力 >= 满战阈值。

**比较约束**：其中一个附加技能等级大于另一个 → 添加比较约束，比较两个依赖目标的技能等级属性。

**掉落约束（全局）**：定义在项目上，例如宝石每日掉落120个 → 所有依赖宝石的目标共享此配置。

**数量约束**：需要衣服和裤子两个部件 → 添加两个依赖约束即可。

#### 关系

一个约束必须 **属于且仅属于** 一个所有者（项目或目标）。

### 进度追踪设计

#### 进度计算

每个目标自身维护进度：
- 进度百分比 = (当前拥有数量 / 所需数量) * 100%
- 当所需数量为 0 或 1 时，进度是二元的（未完成/已完成）

完成判定：
1. 当前目标自身进度 100%
2. 目标上所有约束都满足
3. 所有依赖约束中的依赖目标都已完成

整体项目完成率 = 已完成目标数 / 总目标数

#### 进度历史

为了支持进度变化追踪，可以扩展进度历史记录：
- 记录每次进度更新的时间和变化量
- 可用于分析养成速度和预测完成时间

### 资源掉落配置

游戏中资源掉落通常有每日/每周限制，可以将这些信息记录在目标上或作为扩展属性。

#### 掉落约束

资源掉落限制使用项目级别的掉落约束定义：

- **掉落约束类型**：定义资源每日/每周获取上限
- **参数结构**：{ resourceId: 资源目标ID, daily: 每日数量, weekly: 每周数量 }
- 定义在项目级别，所有需要该资源的目标共享配置

**优势**：统一管理全局掉落，修改时只需修改一处。

#### 预计完成时间计算

基于资源掉落限制自动推算预计完成时间：
- 对于每个资源，查找项目级别的掉落约束
- 缺口 = 所需数量 - 当前拥有
- 根据每日/每周掉落量计算所需天数
- 对于依赖多个资源的成品目标，取最大所需天数作为预计完成时间
- 支持用户手动调整预计时间

**复杂情况处理**：
- 当一个武器需要100个a宝石和100个b宝石，且宝石总掉落每日120个
- 可以添加项目级别的约束，约束宝石总数每日获取不超过120
- 系统根据分配比例计算各资源所需天数

### 标签系统

标签作为独立系统存在：
- 标签可以跨项目使用
- 支持对目标进行多维度分类
- 标签包含名称和颜色属性

### 数据关系图

```
Project (1) ───────────┐
                       ├─> Goal (0..N)
                       │
                       └─> Tag (0..N) ←─┐
                                       │
               Goal (1) ────────────┐   │
                                  │   │
                                  ├──> Constraint (0..N)
                                  │
Constraint (0..N) ───────────────────> Goal (0..N)  ← 依赖关系

        ↑
        │
Project (1) ────────────────────> Constraint (0..N)  ← 全局约束
```

架构保持极简：只有 **项目**、**目标**、**约束** 三个核心概念。  
约束可以属于项目（全局约束）或目标（目标约束）。

## 技术选型：数据库存储

建议直接使用关系型数据库存储数据：

### 优势

1. **数据持久化**：自动保存所有数据，无需担心丢失
2. **复杂查询支持**：可以方便查询"需要100个a宝石且当前不足"等复杂条件
3. **级联删除**：数据库外键自动处理级联删除
4. **事务支持**：批量操作保证数据一致性
5. **易于查询**：SQL可直接查询各种统计信息（总缺口、预计天数等）

### 推荐数据库

- **SQLite**：轻量级，单文件，适合桌面应用
- **PostgreSQL**：功能强大，适合需要多用户访问的场景
- **MySQL**：成熟稳定，社区资源丰富

## 数据操作接口设计

### 项目操作

- 读取目标列表（可按项目筛选，可按优先级排序）
- 读取单个目标详情（包含所有材料需求和特殊条件）
- 更新目标信息
- 更新材料拥有数量
- 重新计算进度
- 更改目标状态
- 删除目标（级联删除下属所有约束）

### 约束操作

- 添加约束到所有者（项目或目标）
- 更新约束参数
- 删除约束
- 检查约束是否满足
