# 插件开发完整指南

## 概述

本项目采用插件化架构设计，核心系统只包含项目、目标、约束三个基础模块。所有非核心功能都通过插件方式扩展。一个完整的插件需要同时包含后端和前端两部分。

## 插件开发前置知识

### 设计原则

- 核心系统保持稳定，不修改核心表结构
- 通过新增数据表扩展功能
- 保持与核心表的外键关联
- 插件可独立启用或禁用
- 遵循开闭原则，对扩展开放，对修改关闭

### 技术栈

后端使用：
- TypeScript + Node.js
- Express 框架
- Prisma ORM
- RESTful API 设计

前端使用：
- React + TypeScript
- Vite 构建工具
- Tailwind CSS
- shadcn/ui 组件库

## 后端插件开发

### 目录结构

每个后端插件独立存放在单独目录中，包含数据模型、路由、控制器、服务等模块。

### 开发步骤

#### 1. 创建插件目录

在后端源码的插件目录下创建插件文件夹，使用小写字母和连字符命名。

#### 2. 定义数据模型

如果插件需要存储自定义数据，在 Prisma schema 中定义新表。表名使用插件前缀，避免命名冲突。

表设计需要满足：
- 添加必要字段
- 与核心表建立外键关联
- 设置级联删除规则
- 添加创建时间和更新时间字段

#### 3. 创建数据访问层

实现基础的 CRUD 操作，封装数据库访问逻辑。按照存储库模式组织代码，分离数据访问和业务逻辑。

#### 4. 实现业务逻辑

在服务层编写业务逻辑代码，处理具体的业务流程。调用数据访问层获取数据，处理业务规则后返回结果。

#### 5. 定义 API 路由

在路由文件中定义插件的所有 API 端点，遵循 RESTful 设计风格。每个端点对应到控制器方法。

#### 6. 实现控制器

编写控制器方法处理 HTTP 请求，调用服务层处理业务，返回 JSON 响应。统一处理错误和异常。

#### 7. 注册路由

在主路由文件中导入并注册插件路由，设置统一的路由前缀区分不同插件。

### 后端最佳实践

- 使用类型推断，减少不必要的类型注解
- 保持函数单一职责
- 错误处理统一交给上层
- 遵循项目代码风格指南
- 尽可能利用现有基础设施

## 前端插件开发

### 目录结构

前端插件独立存放，包含组件、页面、API 调用等模块。每个插件一个目录，保持结构清晰。

### 开发步骤

#### 1. 创建插件目录

在前端源码的插件目录下创建插件文件夹，命名与后端保持一致。

#### 2. 定义 API 客户端

在 API 目录中定义调用后端插件接口的方法，封装 HTTP 请求。统一处理请求和响应格式。

#### 3. 开发组件

按照功能需求开发 React 组件，使用项目已有组件库保持风格一致。拆分小组件，提高复用性。

#### 4. 创建页面

如果插件需要独立页面，创建页面组件并整合到路由系统。页面负责数据获取和状态管理。

#### 5. 集成到界面

根据插件功能需求，集成到合适的位置。可以是独立页面、侧边栏、模态框等形式。

#### 6. 添加样式

使用 Tailwind CSS 编写样式，遵循项目设计规范。保持与现有界面风格协调统一。

### 前端最佳实践

- 组件拆分合理，避免过大组件
- 使用 TypeScript 类型定义保证类型安全
- 合理管理状态，避免不必要的重新渲染
- 复用项目已有组件和工具函数
- 保持界面风格一致性

## 插件集成流程

### 数据库迁移

开发完成后生成迁移文件，执行迁移更新数据库结构。检查外键约束和索引是否正确创建。

### 路由注册

后端在主路由文件中注册插件路由，前端在路由配置中添加插件页面路由。确保路由命名空间不冲突。

### 菜单集成

如果需要在导航菜单中显示，添加菜单项配置。设置正确的路由路径和图标。

### 测试验证

完成集成后进行完整测试：
- 验证数据库表正确创建
- 验证 API 接口正常响应
- 验证前端页面正常显示
- 验证交互功能正常工作
- 验证删除操作级联正确

## 轻量级插件开发

对于简单功能，可以直接利用核心系统的约束表存储数据，不需要新建数据表。这种方式开发更快，更轻量。

### 适用场景

- 配置型功能
- 参数化规则
- 简单条件限制
- 标签分类功能

### 开发方式

- 在约束表中存储插件数据
- 使用 param 字段存储 JSON 格式参数
- 约束类型区分不同插件
- 通过所属目标ID关联数据

这种方式不需要修改数据库结构，开发部署更快捷。复杂功能再考虑新建表。

## 代码风格规范

### 命名规范

- 文件名使用小写字母和连字符
- 类名使用 PascalCase
- 变量和函数使用 camelCase
- 常量使用 UPPER_SNAKE_CASE

### 代码组织

- 导入语句放在文件顶部
- 类型定义放在前面
- 实现逻辑放在后面
- 导出放在文件末尾

### 注释规范

- 每个导出函数需要添加函数级注释
- 说明函数功能和参数含义
- 复杂逻辑添加说明注释
- 避免显而易见的注释

## 常见问题

### 命名冲突

插件表名添加插件前缀，避免与核心表和其他插件表冲突。路由路径也使用插件前缀。

### 数据一致性

始终建立正确的外键约束，依赖数据库保证数据一致性。删除操作设置正确的级联规则。

### 向后兼容

新增功能保持向后兼容，不删除已有字段。数据迁移处理好存量数据。

### 性能优化

合理添加索引，避免 N+1 查询问题。前端合理缓存数据，减少不必要的请求。
