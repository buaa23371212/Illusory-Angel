# 闹钟提醒插件开发文档

## 插件功能概述

为目标添加闹钟提醒功能，用户可以为单个目标设置提醒时间，到时间后触发通知提醒。该插件利用核心系统的约束表存储数据，不需要新建数据表。

## 设计思路

利用约束表存储闹钟数据，每个闹钟对应一条约束记录：
- 所有者类型为 `GOAL`，关联到具体目标
- 约束名称固定为 `alarm_clock`
- params 字段存储闹钟的具体配置，结构固定语义清晰

这种设计不需要修改数据库结构，开发部署简单，符合轻量级插件开发规范。

## 约束数据结构

### 约束基本信息

- **ownerType**: `GOAL`（固定）
- **ownerId**: 目标ID（关联到具体目标）
- **constraintName**: `alarm_clock`（固定，表示这是闹钟提醒）

### params JSON 结构

params 字段是一个 JSON 对象，包含以下字段：

- **enabled**: 布尔值，是否启用提醒
- **time**: 字符串，提醒时间，使用 HH:mm 格式（24小时制），例如 "08:00"
- **days**: 数组，每周哪些天数提醒，0 表示周日，1-6 表示周一到周六，例如 [1,2,3,4,5] 表示工作日
- **message**: 字符串，自定义提醒消息，可为空
- **lastTriggered**: 整数，上次触发时间戳，用于记录上次提醒时间

### 示例结构

一个完整的 params JSON 示例结构如下：

{
  "enabled": true,
  "time": "08:30",
  "days": [1, 2, 3, 4, 5],
  "message": "该起床做养成任务了",
  "lastTriggered": 1718400000000
}

## 后端 API 设计

### 接口端点

所有接口统一使用前缀 `/api/v1/goals/:goalId/alarm`

- **GET** `/api/v1/goals/:goalId/alarm` - 获取目标的闹钟配置
- **POST** `/api/v1/goals/:goalId/alarm` - 创建或更新闹钟配置
- **DELETE** `/api/v1/goals/:goalId/alarm` - 删除闹钟配置
- **GET** `/api/v1/alarm/due` - 获取当前需要触发的所有闹钟（轮询接口）

### 请求响应格式

遵循项目统一的 RESTful 响应格式，使用标准的成功失败标记。

## 功能需求

### 基本功能

- 为目标设置闹钟提醒
- 修改已有的闹钟配置
- 删除闹钟提醒
- 查询闹钟配置
- 定时检查需要触发的提醒
- 触发提醒后返回闹钟列表给前端

### 业务规则

- 一个目标只能有一个闹钟配置
- 时间格式必须为 HH:mm 24小时制
- 天数范围必须在 0-6 之间
- 不启用的闹钟不会触发提醒
- 同一天同一闹钟只触发一次

## 前端功能需求

### 页面组件

在目标详情页面集成闹钟配置面板，用户可以：
- 开启/关闭闹钟
- 选择提醒时间
- 选择每周重复天数
- 输入自定义提醒消息

### 轮询机制

前端定时轮询检查是否有需要触发的闹钟，如果有则显示通知提醒用户。

### 交互设计

- 闹钟已启用显示醒目标识
- 点击进入编辑模式
- 保存后即时生效
- 删除后清空配置

## 后端目录结构

插件遵循项目插件目录规范，后端存放位置：

`src/plugins/alarm-clock/`

目录结构：
- `routes.ts` - 路由定义
- `controller.ts` - 控制器
- `service.ts` - 业务逻辑（检查闹钟触发等）
- `types.ts` - 类型定义

因为闹钟插件有定时检查和触发的业务逻辑，所以需要 Service 层处理这些复杂逻辑。

## 类型定义

在 `types.ts` 中定义闹钟配置的 TypeScript 接口，与 params JSON 结构对应，保证类型安全。

## 业务逻辑处理

### 获取闹钟配置

通过约束表查询，找到目标下名称为 `alarm_clock` 的约束，解析 params 得到配置。

### 保存闹钟配置

如果已存在则更新，不存在则创建。保存时验证时间格式和天数范围。

### 删除闹钟配置

删除对应的约束记录。

### 检查待触发闹钟

查询所有启用的闹钟配置，判断当前时间是否匹配时间和星期，如果匹配且今天未触发过，则返回给前端触发通知。

触发后更新 lastTriggered 记录当前时间戳，避免重复触发。

## 集成位置

### 布局结构

当前系统布局参考项目详情设计，修改方案如下：
- 在目标卡片上添加更多操作下拉菜单（与删除按钮放在同一工具栏）
- 用户点击"目标详情"选项后，在web右侧内容页展示目标详情
- 闹钟配置面板作为目标详情页的一个配置区域

### 具体集成位置

- **操作入口**：目标卡片右侧操作工具栏添加"目标详情"选项（放在删除按钮旁边，使用下拉菜单折叠）
- **详情展示**：点击后在右侧主内容区域展示目标详情页
- **配置面板**：闹钟配置面板集成在目标详情页中，作为可配置属性区域
- **轮询位置**：前端应用主循环定时轮询检查待触发闹钟，收到需要触发的闹钟后，弹出通知提醒用户

## 权限与数据安全

- 用户只能操作自己项目下目标的闹钟（当前单用户系统不需要额外权限控制）
- 输入数据需要验证格式合法性
- 时间格式验证确保为有效的 HH:mm 格式
- 天数范围验证确保在 0-6 范围内

## 轮询策略

前端轮询间隔设置为一分钟检查一次，既能保证及时提醒，又不会对服务器造成过大压力。如果一分钟内有多个闹钟同时到达，可以一起触发通知。

## 浏览器通知权限

前端需要请求浏览器通知权限，获得权限后才能弹出系统通知。如果用户拒绝授权，只在页面内显示提醒。

## 数据迁移

本插件使用核心约束表，不需要新建表，不需要数据库迁移。开发完成直接部署使用。

## 完成验收标准

- 可以成功为目标创建闹钟配置
- 可以修改已有的闹钟配置
- 可以删除闹钟配置
- 到时间后正确触发提醒
- 同一天同一闹钟不会重复触发
- 不启用的闹钟不会触发
- 类型定义完整
- 代码风格符合项目规范
