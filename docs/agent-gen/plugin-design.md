# 插件设计文档

## 设计理念

核心系统保持最小化，只包含项目、目标、约束三个基础表。所有非核心功能通过插件方式扩展，插件可以在需要时逐步加入系统。

插件扩展遵循开闭原则，不修改核心表结构，通过新增数据表实现功能扩展。

## 插件扩展规范

### 扩展原则

所有插件都通过新增数据表扩展功能，不修改核心表结构。保证核心系统稳定性。

### 外键约束

插件表与核心表保持外键关联，确保数据一致性。删除目标时级联删除插件数据。

### 独立部署

每个插件独立设计，可以按需启用或不启用，不启用不占用存储空间。

### 利用约束系统定制

插件可以利用核心系统的约束表存储自定义约束数据，通过灵活的 `param` 字段，实现各种个性化功能。约束系统为插件提供了统一的数据存储基础设施，插件无需额外建表即可实现诸如前置依赖、数量要求、条件判断等定制逻辑。

这种方式使得插件可以快速迭代，轻量级功能直接使用约束系统，复杂功能再新增数据表，灵活平衡了简洁性和扩展性。

## 插件架构设计

### 架构选型：静态插件架构

本项目采用**静态插件架构**，类似于 Go 语言的模块化设计，而非 VSCode 风格的动态插件系统。这种设计选择基于以下考虑：

1. **简洁性优先** - 对于个人项目，静态架构足够使用，实现简单，维护成本低
2. **编译时检查** - 所有插件在编译时接入，TypeScript 类型检查覆盖所有插件代码，减少运行时错误
3. **无需运行时加载** - 不依赖动态代码加载，避免沙箱隔离、安全检查等复杂性
4. **独立开发体验** - 每个插件有独立目录，代码隔离清晰，便于开发和维护

### 与 VSCode 风格动态插件的对比

| 特性 | 当前静态架构 | VSCode 风格动态插件 |
|------|-------------|-------------------|
| 动态安装卸载 | ❌ 需要重新编译 | ✅ 支持 |
| 独立发布版本 | ❌ 随主版本更新 | ✅ 支持 |
| 插件市场 | ❌ 不支持 | ✅ 支持 |
| 沙箱隔离 | ❌ 不隔离 | ✅ 隔离运行 |
| 类型安全 | ✅ 编译时检查 | ⚠️ 运行时检查 |
| 实现复杂度 | 低 | 高 |
| 适合场景 | 个人项目、内核扩展 | 通用平台、第三方插件生态 |

## 插件启用与禁用

### 静态架构下的启用/禁用机制

在静态插件架构中，插件的启用和禁用是在**编译时**通过代码方式控制的，而非运行时动态切换：

#### 前端插件启用/禁用

前端插件统一在 `web/src/plugins/index.ts` 中注册：

```typescript
// 导入插件
import statisticsPlugin from './registered/statistics';
import alarmClockPlugin from './registered/alarm-clock';

// 注册插件
// 要禁用某个插件，只需注释掉对应的注册行即可
export const registeredPlugins = [
  statisticsPlugin,         // 启用：统计概览插件
  alarmClockPlugin,        // 启用：闹钟提醒插件
  // myNewPlugin,           // 禁用：注释掉即表示禁用，不参与编译
];
```

**禁用插件效果**：
- 插件代码不会被打包到最终产物中
- 插件注册的所有扩展点（导航菜单、内容面板等）都不会显示
- 不影响其他插件的正常运行
- 数据库不会创建该插件的表（如果插件有数据表的话）

**启用插件**：取消注释即可，重新编译后生效。

#### 后端插件启用/禁用

后端插件通过模块导入方式控制，在入口文件中注册：

```typescript
// backend/src/plugins/index.ts
import { statisticsPlugin } from './statistics';
import { alarmClockPlugin } from './alarm-clock';

// 注册插件，注释掉即可禁用
export const plugins = [
  statisticsPlugin,         // 启用：统计插件
  alarmClockPlugin,        // 启用：闹钟插件
  // myPlugin,             // 禁用：注释掉即禁用
];
```
